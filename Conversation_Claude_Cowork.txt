🟡 LEVEL 2 — Minimal, production-safe design
Guiding principles
Additive only. No column drops, no FK changes, no semantic shifts on existing tables. Every existing query keeps working unchanged.
Enrollment is the source of truth for "what program this student was in this year." Program is an attribute of an enrollment, not a mutable attribute of the student.
Immutability via append-only. History is not edited; new states create new rows. Mirrors the existing Enrollment.status = inactive pattern instead of deleting rows.
Event log for traceability. A separate table records changes (who, when, why, from→to) so we can answer "why did this student switch programs?" without reverse-engineering enrollment timestamps.

All under /api/v1/, consistent with existing routers. Permissions follow the PERM-ERP:* pattern already in use.

Program catalog

GET    /programs — list active programs for the school. Roles: ADM, TCH, parents (read).

PATCH  /programs/{program_id} — edit name/description/active flag. Permission: PERM-ERP:program:manage.
DELETE /programs/{program_id} — soft-delete by setting is_active = false (never hard delete; it would orphan history).
Program assignment (writes)

POST /enrollments/{enrollment_id}/program — set/replace program_id on an enrollment.
Body: { program_id, reason_code, reason_note? }
Permission: PERM-ERP:enrollment:assign (reuse).
Side effect: writes one ProgramAssignmentEvent row.
Idempotency: same headers as existing write endpoints.
Optionally, augment the existing POST /enrollments body to accept an optional program_id so initial enrollment can include the program in one call. (Backward-compatible: field is optional.)

Program assignment (reads)

GET /students/{student_id}/program-history — full chronological program history for a student. Returns events ordered by occurred_at DESC. Each entry: { academic_year, period, from_program, to_program, reason_code, actor, occurred_at }.
GET /students/{student_id}/academic-timeline — richer view per academic year, joining Enrollment + Class + Program + AcademicYear. Each entry: { academic_year, period, class_name, program: { id, code, name }, status, started_at, ended_at }. This is the canonical "transcript header" view.
GET /students/{student_id}/current-program — convenience: latest active enrollment's program for the current academic year.
Permissions: students read their own; parents read linked children (via existing parent_child_links); ADM/TCH read scoped by school + permission.

L2.4 — Data flow
Initial assignment (new enrollment with a program)

Admin calls POST /enrollments with program_id.
Service validates: student & class & period exist, same school; program exists, same school, is_active.
Insert Enrollment(..., program_id=P) (the existing INV-ERP-CLASS-ACTIVE invariant continues to apply).
In the same transaction, insert ProgramAssignmentEvent(reason_code=INITIAL, from=NULL, to=P, …).
Return enrollment + the event id.
Program change mid-year (transfer between filières)

Admin calls POST /enrollments/{id}/program with new program_id and reason_code=TRANSFER.
Service does not mutate the previous enrollment's program field destructively. Two valid implementations — pick one in the implementation phase:
(a) Soft-replace pattern (recommended): mark current enrollment inactive, create new Enrollment row with the new program_id (and possibly a new class_id), write event with from_enrollment_id and to_enrollment_id. This preserves the historical fact of the old program for that period.
(b) In-place update pattern: update program_id on the existing row but always write the event. Simpler, but loses period-level historical fidelity.
Recommendation: (a) because it makes history reconstructable purely from Enrollment rows, and matches the existing soft-delete idiom (status = inactive).
Year-rollover promotion (program continuity or transition)

Handled by the same flow as initial assignment, in a new period of a new AcademicYear. reason_code = PROMOTION. The previous year's enrollment naturally remains in the database, untouched.
Reading history

program-history: query program_assignment_events filtered by student, ordered by occurred_at. Joins to programs for human-readable names. O(events) per student, paginated.
academic-timeline: query enrollments join classes join academic_years join programs, filtered by student, ordered by (academic_year.date_start, period.date_start). This is the "transcript view".

Web admin (existing React/Next stack): new "Programs" CRUD screen under the school admin area; an "Assign program" affordance on the existing enrollment screen with a modal collecting program_id + reason_code + reason_note. 🔴 LEVEL 3 — Full academic lifecycle (only if it pulls its weight)
What L3 adds, conceptually
feature	what it gives you	what it costs
Program versioning (curriculum evolves)	Track which version of "Sciences Maths 2024" a student studied under	New table program_versions; FK on enrollment moves to the version, not the program; admin UX to publish a version; non-trivial migration story for existing enrollments.
Program equivalences	"2023 SM v1 ≈ 2025 SM v2 for transcript purposes"	New table program_equivalences; equivalence resolution logic in transcript service.
Unified promotion + transition model	One AcademicTransition event covering same-year transfer, year-rollover, repeating, dropping, returning	A unified state machine (real complexity); replaces or wraps ProgramAssignmentEvent.
Academic snapshots per year	A frozen, point-in-time JSON document of "everything about this student this year" — protects transcripts from later schema changes	New academic_snapshots table (JSONB blob); snapshot job at year-close; storage growth ~O(students × years).
Cross-year transcript generator	Single document showing all years, programs, classes, grades, attendance summary	Significant service-layer work; must reconcile equivalences and program versions; PDF generation and i18n.
Rules engine for eligibility	"Can student X promote to program Y in year Z?" — declarative rules	Either a small DSL or a hand-coded service with rules tables; tests get expensive.
L3 risks (worth being honest about)
Program versioning is a 2× complexity multiplier on every read path. Today you join Enrollment → Program. Tomorrow it's Enrollment → ProgramVersion → Program, plus equivalence resolution. Every report, every analytics query, the mobile app — all of them get slightly harder. You should only pay this cost if the school's curriculum genuinely changes meaningfully year-over-year and that change matters to consumers of the data.
Snapshots create dual sources of truth. "Is the snapshot the truth, or are the live tables?" This must be answered explicitly per use case (transcripts: snapshot wins; current-state queries: live tables win). Otherwise debugging becomes painful.
Rules engine is the classic over-engineering trap. Eligibility rules tend to be if level == terminale and average >= 10 then allow_promotion. A handful of ifs in a Python service are usually clearer than a DSL. Build the DSL only when you have ≥10 rules that change without code deploys.
Equivalence semantics are politically loaded. Who decides that "Old Math Track ≈ New Sciences Maths"? It's a pedagogical and sometimes regulatory decision (MEN). Encoding it in your DB means enforcing it in the product.
L3 simplifications you can adopt à la carte
Lightweight versioning instead of a full table: add program_version_label (string) and program_version_effective_from (date) directly on programs. No new table, no equivalence model. Lets you say "the SM program changed in 2025" without modelling every version's curriculum tree.
Deferred snapshots: don't build a snapshot table now. Build the transcript generator on top of live tables. Add snapshots only when (a) transcripts must be reproducible byte-identical in court/audit settings, or (b) live-table reads become too slow to recompute on demand.
Promotion as a reason_code, not a separate model. The L2 ProgramAssignmentEvent table already has reason_code = PROMOTION. That's a unified model — just under a less ambitious name. You may not need an AcademicTransition table at all.
Verdict on L3
For a system that's still in active development and where the existing enrollment model has no program field at all, most of L3 is overkill today but none of it is wasted eventually. The honest call is: ship L2 cleanly, then graduate to L3 features one at a time as concrete pain emerges. The L2 design is intentionally compatible with this — program_id on the enrollment can later become program_version_id via a migration that adds the version table and rewrites the FK target, with no API contract break (responses already include { program: { id, code, name } }, which the version table can populate identically).

🧠 Final recommendation
Recommended approach: Hybrid — Level 2 fully, plus the L3 lightweight versioning shim
L2 is the right scope for production now. The single L3 feature worth folding in early is the lightweight version label on programs (version_label, effective_from) — adding it now costs nothing and gives you a cheap forward-compatibility path to full versioning later. Skip snapshots, equivalences, and rules engine until concrete pain forces them.
